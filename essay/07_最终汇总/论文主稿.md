## 题目
基于Web的个性化博客推荐系统设计与实现

## 中文摘要
面向技术博客场景的内容平台通常同时具有知识沉淀、持续阅读和多次回访等特点，单纯依赖时间排序或关键词检索，难以在文章规模增加后持续满足用户的内容发现需求。围绕这一问题，本文结合真实项目，设计并实现了一套基于用户行为分析的个性化技术博客推荐系统。系统采用前后端分离架构，前端使用 Vue 3、Vite、Element Plus 和 Axios 组织页面与交互，后端使用 Spring Boot、MyBatis 与 MySQL 完成业务处理和数据持久化。系统围绕用户注册登录、博客录入与管理、文章检索与详情查看、阅读记录、点赞收藏待读拉黑、评论评分、消息通知、个人中心、兴趣画像展示和个性化推荐等核心功能构建完整业务闭环。

在推荐实现方面，本文没有采用复杂训练流程，而是结合真实工程条件，构建了以用户行为和标签画像为基础的混合推荐链路。系统以浏览记录、阅读时长、点赞、收藏、评分和拉黑等行为信号为输入，通过标签汇总形成兴趣画像，再结合协同召回、标签召回、热门内容兜底和多样性重排生成推荐结果，同时为部分结果提供推荐理由，以增强结果的可理解性。除推荐链路外，系统还在文章录入阶段引入 AI 摘要与标签生成功能，并通过降级逻辑保证在外部接口异常时系统仍可保持基本可用。

在验证环节，本文基于本地工作副本完成了前端生产构建、后端基础上下文加载测试、接口联调与数据库回查。测试结果表明，系统在注册登录、文章发布与搜索、详情查看、阅读时长记录、点赞收藏待读拉黑、评论评分、通知已读处理和推荐结果返回等核心流程上能够完成预期业务闭环；在轻量行为用户场景下，推荐结果能够稳定返回，并通过热门内容兜底保证推荐页可用性。与此同时，本文也明确指出当前系统尚未开展性能测试和推荐量化评估，页面截图与端到端交互证据仍有补强空间，AI 精排也未作为默认主链启用。

总体来看，本文完成了一个真实可运行的个性化技术博客推荐系统的设计、实现整理与工程验证工作。研究结果表明，在本科毕业设计的工程边界内，以行为数据、标签画像和轻量混合推荐为主线，可以较为稳定地构建技术博客场景下的内容管理与个性化分发闭环。

## Title
Design and Implementation of a Web-Based Personalized Blog Recommendation System

## Abstract
Technical blog platforms are different from ordinary content websites because they usually involve long-term knowledge accumulation, repeated visits, and behavior signals that are richer than simple page browsing. When the amount of content continues to grow, chronological listing and keyword retrieval alone are no longer enough to help users discover articles that better match their interests. To address this problem, this thesis designs and implements a personalized technical blog recommendation system based on user behavior analysis. The work is grounded in a real project rather than a hypothetical prototype, and every design description in the thesis is constrained by the actual implementation status of the system.

The system adopts a front-end and back-end separated architecture. On the front end, Vue 3, Vite, Element Plus, and Axios are used to organize pages, routing, interaction logic, and request processing. On the back end, Spring Boot, MyBatis, and MySQL are used to implement business logic and data persistence. Around the real project boundaries, the system supports user registration and login, blog publishing and management, article search, article detail viewing, reading history, reading duration accumulation, likes, favorites, to-read actions, blacklist actions, comments, ratings, notification handling, personal center management, profile summary display, radar chart display, and personalized recommendation.

Instead of claiming a complex experimental recommendation model, the thesis focuses on an engineering-oriented recommendation pipeline that can run stably in the current project. The recommendation logic takes reading logs, reading duration, likes, favorites, ratings, and blacklist actions as behavior signals. These signals are aggregated with article tags to build a lightweight user interest profile. Based on this profile, the system combines collaborative recall, tag-based recall, hot-content fallback, and diversity reranking to generate recommendation candidates. For some recommendation results, the system also returns recommendation reasons in order to improve result interpretability. In addition, the system introduces AI-assisted summary and tag generation during article publishing. When the external AI service is unavailable, a fallback strategy is used so that the publishing process can still remain available.

The validation part of the thesis does not include performance benchmarking or quantitative recommendation metrics. Instead, the system is verified through production build checks, back-end context loading tests, interface-level integration tests, and database back-checking in the local working copy environment. The recorded results show that the main business processes can complete their expected flows: user registration and login can be closed successfully, published articles can be searched and viewed, reading duration records can be updated and queried, likes, favorites, to-read actions, and blacklist actions can be written and reflected, comments and ratings can be submitted and read back, notification states can be updated, and recommendation interfaces can return non-empty results with recommendation sources and recommendation reasons. In lightweight behavior scenarios, the recommendation interface still relies noticeably on hot-content fallback, which is consistent with the system’s design for cold-start and weak-profile conditions.

At the same time, the thesis explicitly keeps several boundaries clear. The current project has not carried out throughput testing, response time comparison, or recommendation accuracy evaluation. Metrics such as recall, precision, NDCG, MAP, AUC, or F1 are therefore not used in the thesis. The AI reranking strategy is reserved in the codebase but is not enabled as the default main recommendation chain. The tables named `recommendation` and `user_behavior` exist in the SQL script, but they are not described as core production-chain tables because stable evidence of a full write-read business loop has not been confirmed. The security description is also intentionally narrowed to password encryption and basic access control, rather than being expanded into a complete authorization framework.

Overall, this thesis completes the design explanation, implementation analysis, and engineering validation of a real personalized technical blog recommendation system. The work shows that, within the scope of an undergraduate software engineering graduation project, it is feasible to construct a usable and explainable content management and personalized distribution loop by combining behavior data, tag-based user profiling, hybrid recommendation, and AI-assisted content analysis in a restrained and verifiable way.

# 第1章 绪论

## 1.1 研究背景

技术博客平台本质上属于内容平台的一种，但它与一般资讯平台相比，具有更明显的知识沉淀属性和更长的阅读决策链。用户在这类平台上不只是被动浏览信息，还会围绕文章进行检索、阅读、收藏、评分和二次回访，因此内容管理和内容分发往往需要同时兼顾。国内已有研究表明，推荐系统的发展已经从单一结果匹配逐步扩展到应用场景适配、用户兴趣识别和结果多样性控制等多个层面[1][2][3]。在内容平台侧，个性化推荐也已经进入移动新闻、数字阅读和其他在线内容服务场景，并对信息获取方式产生了持续影响[4][5]。

随着平台内容规模增加，传统按时间排序或关键词检索的方式难以充分满足用户的信息发现需求。尤其在技术博客场景下，同一主题文章可能来自不同作者、具有不同深度和不同适用对象，如果系统不能根据用户近期行为及时调整内容展示顺序，用户就更容易在信息量增加后陷入筛选成本上升的问题。已有综述性研究指出，推荐研究的重点已经不仅是“能否找到相关内容”，还包括如何结合更多辅助信息、如何改善多样性以及如何增强解释性[6][7][8]。这说明内容平台中的推荐问题已经从简单排序问题，逐步演变为一个涉及行为建模、内容理解和结果呈现的综合工程问题。

另一方面，用户画像和行为建模研究也为内容平台的个性化服务提供了方法基础。国内图书情报和阅读平台相关研究表明，用户画像已经从概念描述逐渐走向基于行为数据、标签建模和应用验证的阶段[9][10][11][12]。同时，也有研究尝试从多维属性和群体差异角度刻画用户特征，为后续偏好建模提供更细的分析视角[13]。对技术博客平台而言，用户的阅读时长、点赞、收藏、评分和拉黑等行为虽然看似分散，但它们共同构成了兴趣识别的基础信号。基于这些信号建立轻量、可解释的画像机制，比直接追求复杂模型更符合本科项目的实现边界。

结合当前工程实践的发展趋势，还可以看到推荐系统对语义信息和结果解释的关注正在增强。国外研究已明显关注带有辅助信息的推荐问题，以及混合推荐中的个性化解释能力[7][15]。本文据此做出的判断是：推荐研究正在朝着更重视语义、上下文和解释的方向发展，这也为后续把大模型能力作为辅助增强工具引入内容平台提供了现实背景。但需要明确的是，这类趋势判断并不等同于“本项目已经实现大模型主导推荐”，本文仍以真实工程实现为限。

## 1.2 研究意义

本课题的意义首先体现在工程实践层面。与单纯的博客展示系统相比，个性化技术博客推荐系统需要同时处理账户管理、内容录入、阅读记录、互动行为、通知联动、画像展示和推荐分发等多条业务链。这意味着系统不再只是一个静态内容容器，而是一个围绕“用户—内容—行为—分发”闭环组织起来的工程项目。对于软件工程专业的本科毕业设计而言，这种项目具有较清晰的模块边界、较完整的前后端链路和较强的综合实现价值。

其次，本课题具有一定的应用意义。技术博客场景中的文章通常具有知识密度高、主题差异大、读者目标明确等特点。若仅靠列表浏览和搜索检索，用户在文章数量增多后容易面对筛选成本上升的问题。通过引入阅读记录、点赞、收藏、评分和拉黑等行为信号，系统可以在内容展示层面形成更有针对性的分发结果，并通过推荐理由、兴趣摘要和雷达图等形式增强推荐结果的可理解性。这样的设计虽然不追求复杂算法实验，但能够在实际使用中改善内容发现效率和个人阅读管理体验。

再次，本课题对方法收敛也有实际价值。现有推荐研究和用户画像研究已经提供了较多理论和方法路径[2][3][11]，但在本科项目中，直接照搬复杂研究模型往往既难以复现，也容易脱离真实工程条件。本文选择以混合推荐、热门兜底、画像汇总和 AI 摘要与标签生成等能力为主线，本质上是在真实可实现范围内完成一次较为克制的系统设计与实现。这样的取向更贴近本科毕业论文的要求，也更有利于保证正文内容与真实项目边界一致。

## 1.3 国内外研究现状

### 1.3.1 国内研究现状

从国内研究看，推荐系统相关工作已经形成较为清晰的演进脉络。较早的研究更多关注推荐结果的相关性和误差控制，之后逐步扩展到排序学习、多样性控制和应用场景适配等方向。已有综述研究表明，国内推荐研究已经不再局限于传统协同过滤的单一路径，而是开始从推荐任务、算法结构和系统应用三个层面进行综合梳理[1][2][3]。同时，多样化推荐综述也显示，研究目标已经从“找到最相关内容”延伸到“在相关性之外兼顾结果丰富度和用户体验”[6][8]。

在用户画像与阅读行为方面，国内研究同样积累了较多成果。图书情报领域和阅读平台相关研究已经开始系统讨论用户画像的研究现状、建模路径和应用场景[11][12]。面向读者行为的研究说明，用户画像不再只是静态标签集合，而是可以与阅读轨迹、内容偏好和场景特征结合起来[10]。对本文而言，这类研究的价值不在于直接照搬其方法，而在于说明“基于行为数据构建兴趣画像”已经具备明确的研究基础。

从内容平台实践角度看，个性化推荐已经不局限于电商和视频平台，也逐渐进入移动新闻和数字阅读等内容服务场景[4][5]。这意味着技术博客系统在引入推荐能力时，并不是在完全空白的场景中展开，而是在一个已有较多相似应用背景的方向上做工程化落地。不过，现有国内研究往往更强调算法本身、传播影响或画像建模，对于轻量级前后端分离系统如何把这些能力组织成稳定可运行的业务闭环，讨论仍不充分。

### 1.3.2 国外研究现状

国外推荐研究长期关注如何利用更丰富的辅助信息提升推荐能力。相关综述研究指出，推荐系统已经从单纯依赖评分矩阵，逐步转向结合内容特征、上下文信息和更多侧信息的方向[7]。同时，也有研究围绕异构隐式反馈展开排序优化，说明国外推荐研究对点击、浏览和偏好等非单一显式评分信号的利用更加重视[14]。这说明国外研究在推荐问题的理解上，已经不再把推荐视为孤立的打分预测问题，而更强调多源信息整合。

与此同时，解释性问题也成为国外研究的重要方向之一。有关混合推荐个性化解释的研究表明，推荐结果不仅需要“给出什么”，还需要在一定程度上解释“为什么出现”[15]。另有研究尝试借助知识图谱和规则学习增强推荐解释能力，进一步体现了推荐系统对结构化语义和解释性的重视[16]。对内容平台而言，这一点尤为关键，因为推荐解释往往直接影响用户对系统结果的接受度。本文在系统设计中保留推荐理由展示，正是受这一研究趋势启发，但实现路径仍然收敛为工程上更轻量的结果级解释，而非完整可解释推荐框架。

从国外研究现状还可以看出，推荐系统的发展越来越强调语义理解、侧信息整合和用户决策可解释性。本文据此做出的归纳是：推荐研究已经明显超出传统相关性优化的单一目标，而逐步走向“更丰富的输入信息 + 更清晰的解释能力”的综合优化方向。这一趋势为大模型和更强语义建模能力进入推荐领域提供了现实背景，但在本项目中，这种能力只体现在 AI 摘要与标签生成等辅助环节，并未进入主推荐链路。

### 1.3.3 现有研究的不足与本文切入点

综合国内外研究可以发现，现有工作已经在推荐方法、用户画像和解释能力等方向形成了较丰富的成果，但仍存在两个与本文密切相关的缺口。第一，很多研究更偏方法综述或算法讨论，对轻量级内容平台如何在真实业务链路中把账户、内容、行为、通知和推荐组织成一个完整系统，描述仍然不足。第二，已有研究常使用复杂实验设计或大规模数据验证方法，而本科项目更需要在真实可实现范围内构造一个功能闭环明确、边界清晰且能够落地验证的系统。

本文的切入点正是在这里。与强调量化指标优化的研究不同，本文围绕一个真实技术博客系统展开，重点解决博客内容录入、阅读行为记录、互动反馈、个人中心展示和个性化推荐的工程实现问题。在推荐部分，本文没有追求复杂训练模型，而是采用混合推荐、热门兜底、推荐理由展示和 AI 录入辅助等较为克制的实现路线，使项目能够在本科毕业论文的范围内保持真实性、完整性和可答辩性。

## 1.4 本文主要研究内容

本文围绕真实项目“个性化技术博客推荐系统”的设计与实现展开，主要研究内容可以概括为四个方面。

第一，围绕系统基础业务闭环，完成用户注册登录、博客录入与管理、文章检索与详情查看、评论评分和个人中心等基础功能设计与实现。第二，围绕用户行为数据沉淀，完成阅读记录、阅读时长、点赞、收藏、待读和拉黑等行为采集机制设计，使用户后续的画像展示和推荐分发具有真实数据基础。第三，围绕个性化分发能力，设计并实现基于行为信号和标签画像的混合推荐链路，并加入热门内容兜底和推荐理由展示，以保证推荐结果的可用性与可解释性。第四，围绕智能分析辅助，在文章录入阶段引入 AI 摘要与标签生成能力，并通过降级逻辑保证该能力在外部接口异常时仍不破坏系统基本可用性。

需要强调的是，本文所有研究内容都以真实项目实现为边界。对于性能测试、推荐量化评估、大模型主导重排序、推荐结果持久化闭环等未被项目真实证实的内容，本文均不作为既成成果展开。

# 第2章 需求分析

## 2.1 系统建设目标

本课题面向的不是通用资讯门户，而是具有技术内容积累、用户阅读行为沉淀和个性化分发需求的博客场景。结合当前项目的真实实现情况，系统建设目标主要集中在三个方面。其一，完成技术博客系统的基础业务闭环，使用户能够完成注册登录、文章发布、文章浏览、文章搜索、评论评分和个人资料维护等常见操作。其二，在基础内容管理之外，引入阅读记录、点赞、收藏、待读和拉黑等行为采集机制，为后续的个性化推荐提供可用的行为依据。其三，在不脱离本科工程实现边界的前提下，将推荐、推荐理由展示、AI 摘要与标签生成等能力整合到现有博客系统中，使系统从单纯的内容展示工具扩展为兼顾内容管理与个性化分发的应用系统。

从当前代码与数据库结构来看，系统的建设范围已经覆盖用户账户管理、博客内容管理、互动行为管理、阅读记录、消息通知、个人中心、个人维度统计展示和混合推荐等模块，但并未扩展为独立后台管理平台，也未开展性能测试和推荐量化评估。因此，本章中的需求分析将以当前已实现功能和真实工程约束为准，不将开题阶段的预期目标直接视为系统现状。

## 2.2 可行性分析

### 2.2.1 技术可行性

系统采用前后端分离实现方式。前端使用 Vue 3、Vite、Element Plus、Axios 和 ECharts 组织页面交互、路由导航、接口请求和图形展示；后端使用 Spring Boot、MyBatis 与 MySQL 完成业务处理与数据持久化。就当前项目结构而言，前端页面已经覆盖登录、注册、首页、详情页和个人中心等主要场景，后端控制层、服务层与数据访问层之间的职责边界也较为清晰，说明该技术路线能够支撑论文所涉及的主要业务需求。推荐模块虽然引入了协同过滤、标签画像、热门兜底和多样性重排等策略，但整体仍建立在现有行为数据和简单可解释逻辑之上，未依赖难以复现的复杂训练流程，因此在本科毕业设计范围内具有较好的可落地性。

### 2.2.2 业务可行性

技术博客场景本身具有明确的业务对象和相对稳定的使用流程。用户需要注册登录后完成文章发布、查阅和互动；系统需要记录阅读行为、展示个人历史与收藏结果，并在内容较多时帮助用户发现更符合兴趣的文章。当前项目已围绕这些核心场景实现相应页面和接口，业务链条比较完整。同时，系统并未引入与主题无关的大规模平台功能，需求范围保持在博客、阅读和推荐三个核心方向内，适合作为本科软件工程项目型论文的研究与实现对象。

### 2.2.3 运行与维护可行性

在运行和维护层面，系统具有较好的基础条件。前端页面按功能划分，主要业务集中在首页、详情页和个人中心；后端采用控制层、服务层、Mapper 层分层组织，数据库核心表围绕用户、博客、评论、通知、阅读记录和互动关系展开。这样的组织方式便于后续定位问题，也便于论文按模块展开说明。需要说明的是，当前系统未形成完整的后台运维平台，也未对部署性能进行专门验证，因此运行与维护可行性更多体现在结构清晰、功能闭环可运行和后续可扩展，而不是体现在运维自动化或性能优化层面。

## 2.3 功能需求分析

### 2.3.1 用户账户与资料管理需求

系统需要支持用户完成注册、登录和资料修改等基础操作。注册阶段应能校验用户名是否重复，避免同名用户产生冲突；登录阶段应能识别错误账号或密码，并对未登录访问进行拦截；资料维护阶段应支持昵称、头像和密码修改，以满足用户个性化使用需要。当前项目中，这些功能均已对应到实际接口和页面实现，因此可作为后续系统实现章节的稳定需求来源。

### 2.3.2 博客内容管理需求

作为博客系统的主体，文章内容管理是核心需求之一。系统需要支持文章发布、文章编辑、文章删除、列表展示、关键词搜索和文章详情查看。在内容录入方面，除了标题和正文外，系统还需要支持标签、封面和摘要等辅助信息，以便后续完成内容展示和推荐处理。当前项目中，若用户未手动填写摘要，后端还会根据正文内容自动截取摘要，这一处理方式虽然简单，但能够满足基础可用性需求。

### 2.3.3 用户互动需求

为了让内容消费行为能够被系统识别并进一步参与推荐，系统需要支持点赞、收藏、待读、拉黑、评论和评分等互动操作。其中，点赞和收藏用于表达正向兴趣，待读用于保留后续阅读意图，拉黑用于显式排除不感兴趣内容，评论与评分则承担互动反馈和文章质量评价功能。当前项目已经通过 `user_like`、`user_action` 和 `comment` 等表实现上述需求，并将部分状态回显到首页、详情页和个人中心页面。

### 2.3.4 阅读行为与个人中心需求

系统不能只停留在“内容可看”，还需要能记录用户如何看。为此，系统需要在文章详情页记录浏览行为、累计阅读时长，并在个人中心中展示浏览历史、我的文章、我的点赞、我的收藏、待读列表、黑名单和消息通知等内容。这样既能提升用户对个人数据的管理效率，也为推荐模块提供更有解释性的行为来源。当前项目已通过详情页进入时增加阅读量、离开时上报阅读秒数、个人中心集中展示等方式满足这一需求。

### 2.3.5 推荐与画像展示需求

当文章数量逐步增加后，仅靠列表浏览和关键词搜索难以满足内容发现需求，因此系统需要具备基础推荐能力。当前项目中的推荐需求主要体现为：首页提供热门内容和猜你喜欢，详情页提供相关推荐，个人中心提供兴趣摘要和标签雷达图展示。需要强调的是，当前项目的推荐能力定位于基于真实行为数据的工程化混合推荐，而非追求复杂算法评测结果。因此，本系统的推荐需求侧重“可运行、可解释、能与现有页面联动”，而不是侧重“量化效果最优”。

## 2.4 非功能需求分析

### 2.4.1 易用性需求

系统面向普通博客使用者，页面交互应当尽量直观。首页需要同时承载文章列表、搜索、热门内容和推荐入口；详情页需要集中展示正文、互动操作、评论区和相关推荐；个人中心需要让用户能够快速切换到消息、书架、浏览历史和资料修改等常用入口。当前项目通过路由组织、来源页返回逻辑和状态回显，已经体现出对基本易用性的考虑。

### 2.4.2 安全性需求

在当前系统边界内，安全性需求主要表现为基础访问控制和敏感信息处理。密码不应明文存储，未登录用户不应直接访问核心页面，文章和评论删除操作应具备作者或管理员校验。需要特别说明的是，当前系统并未实现完整的 Spring Security、JWT 或 RBAC 权限框架，因此本论文中的安全性分析应收敛为密码加密、基础访问拦截和局部权限判断，不应扩展为完整安全体系设计。

### 2.4.3 可维护性需求

可维护性直接影响后续功能扩展和问题定位效率。当前项目采用前后端分离和分层结构，前端页面与后端接口之间的关系相对明确，后端的控制层、服务层和 Mapper 层也具有较好的职责划分。这种组织方式使系统能够按功能模块逐步扩展，也便于在论文中清楚描述各模块之间的调用关系。

### 2.4.4 可扩展性需求

系统在设计上还需要考虑后续扩展空间。当前项目已经为推荐参数配置、AI 摘要生成和 AI 精排扩展策略预留了入口，但其中 AI 精排默认关闭，不应作为当前主链能力表述。对于论文而言，可扩展性需求可以写为“系统保留了后续进一步增强推荐和智能分析能力的接口与结构基础”，而不能写成“系统已经完成全部智能策略落地”。

## 2.5 业务流程分析

### 2.5.1 用户注册登录与访问流程

用户首先通过注册页创建账号，系统对用户名唯一性进行检查，并在保存前对密码进行加密处理。注册成功后，用户通过登录页完成身份验证，登录成功后才能进入首页、详情页和个人中心等核心页面。如果用户在未登录状态下直接访问这些受限路由，前端路由守卫会将其重定向到登录页。该流程保证了后续阅读、互动与个人中心功能都建立在已登录用户基础上。

### 2.5.2 博客发布与浏览流程

用户登录后可在首页发起文章发布操作，输入标题、正文、标签、封面和摘要后提交。若未填写摘要，后端会按正文内容自动生成基础摘要。文章发布成功后会进入文章列表，其他用户可通过首页浏览、关键词搜索或热门内容入口找到该文章，并进一步进入详情页查看正文内容。若文章作者或管理员需要调整内容，还可以在详情页进入编辑流程。

### 2.5.3 互动与通知流程

在文章详情页中，用户可以对文章进行点赞、收藏、加入待读、拉黑、评论和评分等操作。点赞和评论会进一步触发通知逻辑，使文章作者能够在个人中心消息页看到新的提醒；收藏、待读和拉黑则主要影响用户自己的书架状态和推荐过滤结果。该流程体现了互动行为不仅服务于页面反馈，也会影响后续的消息联动与推荐表现。

### 2.5.4 阅读记录与推荐流程

用户打开文章详情页时，系统会增加文章阅读量并记录一次浏览日志；当用户离开页面时，前端会将阅读时长上报给后端。后端再结合阅读时长、点赞、收藏以及评分等行为信号构建用户兴趣画像，并将其用于首页个性化推荐。对于行为较少的用户，系统以热门内容作为兜底；对于已有行为沉淀的用户，系统则根据画像和召回策略生成更贴近其兴趣的候选结果。由此形成“阅读行为采集—画像构建—推荐展示”的业务闭环。

# 第3章 系统总体设计

## 3.1 系统总体架构设计

### 3.1.1 总体架构组成

本系统采用前后端分离结构，整体可划分为前端展示层、后端业务层和数据存储层三个部分。前端负责页面展示、路由切换、用户交互和接口调用；后端负责账户、博客、互动、通知、推荐和 AI 辅助等业务处理；数据库负责持久化存储用户、博客、评论、通知和行为记录等核心数据。这样的组织方式一方面便于将页面交互与业务处理解耦，另一方面也便于在论文中按照模块逐层展开说明。

从真实项目实现看，系统的使用入口主要集中在登录页、注册页、首页、文章详情页和个人中心。用户完成登录后，可在首页完成文章浏览、搜索、发布、热门查看和个性化推荐查看；在详情页完成阅读、互动、评论评分和相关推荐浏览；在个人中心完成历史、收藏、待读、黑名单、消息和画像查看。与之对应，后端按照控制层、服务层和数据访问层组织接口链路，将前端请求转换为数据库读写和业务处理结果。

### 3.1.2 前端架构设计

前端基于 Vue 3 和 Vite 构建，使用 Vue Router 管理页面导航，使用 Axios 统一发起接口请求，使用 Element Plus 实现主要表单与交互组件，并使用 ECharts 完成标签雷达图等可视化展示。从页面组织方式看，系统前端并未拆分为大量独立页面，而是围绕用户进入频率最高的几个核心页面组织功能，减少了本科项目中常见的页面过度分散问题。

在路由设计上，登录页和注册页承担账户入口功能，首页承担文章展示、搜索、推荐和发布功能，详情页承担文章阅读与互动功能，个人中心承担个人数据管理和行为结果回显功能。前端还通过路由守卫对未登录访问进行拦截，使核心业务页面建立在已登录状态基础上。对于推荐结果、互动状态和消息未读数等内容，前端采用接口返回后即时回显的方式处理，不依赖额外的复杂状态管理框架。

### 3.1.3 后端架构设计

后端基于 Spring Boot 构建，并使用 MyBatis 完成数据库访问。整体采用较为清晰的控制层、服务层和 Mapper 层分层结构。控制层负责接收前端请求并组织返回结果，服务层负责封装账户管理、博客管理、评论评分、通知联动、画像计算与推荐生成等业务逻辑，Mapper 层负责具体的 SQL 执行和数据读写。

这种分层方式使系统更适合工程化实现与后续维护。例如，文章增删改查主要由博客控制器和博客服务负责；点赞、收藏、待读、拉黑等操作分别由点赞接口和统一动作接口处理；推荐相关逻辑则集中在推荐画像服务、召回策略和重排策略等类中。虽然系统没有引入完整的安全框架或后台管理平台，但当前结构已经能够支撑博客系统与推荐模块的主要业务需求。

## 3.2 功能模块设计

### 3.2.1 用户与资料管理模块

用户与资料管理模块负责支撑注册、登录、资料修改、密码修改和头像上传等基础功能。该模块的核心任务不是实现复杂权限体系，而是完成博客系统基本可用性的前提条件。系统通过用户表保存账号、密码、昵称、头像和角色信息，通过前端路由守卫和后端局部校验实现基础访问控制。对于论文写作而言，本模块应定位为“账户与资料管理”，而不是扩展为完整的权限管理子系统。

### 3.2.2 博客内容管理模块

博客内容管理模块负责文章录入、文章展示和文章维护。用户可在首页发起文章发布，输入标题、正文、标签、分类、原文链接、摘要等信息后提交保存；其他用户可通过首页列表、搜索结果或热门入口浏览文章，并进一步进入详情页查看完整内容。若当前用户是文章作者或具备管理员角色，还可执行文章编辑和删除。

从实现方式看，该模块不仅负责基本的增删改查，还与 AI 摘要生成和推荐展示产生联动。文章中的标题、正文、标签和摘要既用于前端展示，也为推荐模块提供了内容侧基础信息。因此，在总体设计中，博客内容管理模块既是业务入口，也是后续行为分析和推荐处理的基础数据来源。

### 3.2.3 互动与消息模块

互动与消息模块围绕用户对文章的反馈行为展开，主要包括点赞、收藏、待读、拉黑、评论、评分和消息通知。系统将点赞独立存放在 `user_like` 表中，将收藏、待读和拉黑统一记录在 `user_action` 表中，并通过评论表保存文本评论和评分信息。这样的划分方式既保留了点赞与其他动作在业务语义上的差异，也降低了动作类功能扩展时的结构复杂度。

消息通知主要围绕点赞和评论联动展开。用户对文章执行点赞或评论后，系统会向文章作者写入通知记录，并在个人中心消息页提供未读数统计、单条已读和全部已读处理。该模块的设计重点在于“互动行为与消息反馈的一致性”，而不是构建独立的消息推送平台。

### 3.2.4 个人中心与统计展示模块

个人中心模块用于集中展示用户与系统交互后沉淀下来的数据结果。系统将我的文章、我的点赞、收藏列表、待读列表、黑名单、浏览历史、消息中心和资料设置等功能收纳在同一页面体系中，减少多页面跳转带来的割裂感。对于博客系统而言，这种组织方式能够同时满足内容管理和个人行为回看两类需求。

在统计展示方面，当前系统更接近“个人维度概览”而非独立数据分析平台。系统展示累计阅读时长、兴趣标签摘要和标签雷达图，目的在于让用户看到自己的兴趣变化趋势，并为推荐结果提供解释支撑。因此，本模块的总体设计应收敛为“个人中心与画像展示”，不宜拔高为“数据分析平台”。

### 3.2.5 推荐模块

推荐模块是系统区别于普通博客系统的关键组成部分。当前项目并未采用完全依赖单一算法的实现方式，而是以工程化混合推荐为总体思路，将用户行为、标签画像、热门内容和结果多样性控制组合在同一链路中。首页推荐接口负责输出个性化结果，详情页相关推荐接口负责根据当前文章内容补充相近内容，热门接口则在行为信号不足时承担兜底作用。

推荐模块的设计目标主要体现在三个方面：第一，结合浏览、时长、点赞、收藏和拉黑等真实行为信号生成更贴近兴趣的候选内容；第二，在冷启动场景下仍能保持推荐页可用；第三，通过推荐理由字段提高结果的可解释性。需要明确的是，当前系统虽然预留了 AI 精排策略，但配置中默认关闭，因此总体设计仍以真实启用的混合推荐主链为准。

## 3.3 业务流程与数据流设计

### 3.3.1 博客发布数据流

文章发布流程从前端表单输入开始。用户在首页填写标题、正文、标签、分类、摘要和原文链接后提交给后端，后端博客服务负责校验必要字段并写入博客表。若用户未填写摘要，系统会根据正文自动截取生成基础摘要；若用户手动触发 AI 分析，则后端会调用 AI 接口生成摘要和标签建议，再回填到录入流程中。文章保存成功后，前端重新拉取文章列表，形成“录入—保存—回显”的基本数据流。

这一流程的设计重点不在复杂审核机制，而在保证录入字段完整、页面反馈及时和后续推荐可复用。文章的标签、摘要和正文内容既服务于展示，也为后续相关推荐和个性化推荐提供内容依据。

### 3.3.2 阅读行为数据流

阅读行为数据流从用户进入详情页开始。系统在用户打开文章详情时增加阅读量，并写入一次浏览日志；当用户离开详情页时，前端将本次阅读停留时间提交给后端，由后端更新阅读时长。最终形成的行为结果存入 `visit_log` 表，并在个人中心中以浏览历史或累计阅读时长的形式回显。

与只记录一次点击不同，当前系统将阅读时长也纳入数据链路，使浏览行为不再只是“看过”或“没看过”的二元关系。虽然本文不对时长权重展开量化评估，但从总体设计上看，将阅读时长纳入画像与推荐输入，能够让系统对用户阅读深度形成更细的区分。

### 3.3.3 用户互动数据流

用户互动数据流主要包括点赞、收藏、待读、拉黑、评论和评分六类行为。点赞操作会更新点赞关系，并同步影响文章点赞状态回显；收藏、待读和拉黑通过统一动作接口写入动作表，再在个人中心对应列表中回显；评论和评分提交后写入评论表，并触发文章平均分的重新计算。部分互动行为还会进入消息通知链路，使作者在消息页中看到新的点赞或评论提醒。

这一设计的价值在于，同一类用户行为不会只停留在某一处页面状态中，而会同时影响文章展示、个人中心、通知中心和推荐输入。由此，互动功能从“页面按钮”转变为“全链路行为信号”。

### 3.3.4 推荐结果生成数据流

推荐结果生成流程以行为数据为起点。系统首先汇总浏览记录、阅读时长、点赞、收藏以及评分等信号，计算用户兴趣标签及其权重，并据此构建用户画像。随后，推荐引擎分别通过协同过滤召回、标签召回和热门兜底召回获得候选内容，再通过多样性重排控制同作者和同主标签的过度集中，最后为结果附加推荐来源和推荐理由信息。

在前端展示层，首页推荐结果不是一次性固定写死，而是按候选池分批展示，用户可以继续切换推荐批次。对于行为数据较少的用户，系统直接回退到热门内容或冷启动策略，以避免推荐页出现空白。由此形成“行为采集—画像构建—多路召回—重排解释—页面展示”的推荐数据流。

## 3.4 数据库设计

### 3.4.1 数据库总体设计原则

数据库设计围绕博客系统的核心业务闭环展开，重点服务账户管理、文章管理、互动反馈、通知联动和阅读行为记录五类需求。就当前项目来看，数据库并未追求复杂的范式拆分，而是采用更适合本科工程实现的结构：在保证主要实体独立的前提下，通过少量冗余字段降低查询复杂度，提高页面联调的直接性。

例如，评论表中保留评论人昵称和头像，通知表中保留触发者昵称和通知摘要，这些设计都体现了当前项目更强调“页面使用便利”而非“极致数据库规范化”。这一点也决定了本章数据库设计应以支撑真实功能为核心，而不是追求复杂建模本身。

### 3.4.2 核心实体表设计

`user` 表是用户账户管理的核心表，主要保存用户名、密码、邮箱、昵称、头像和角色等信息，其中用户名设置唯一约束，用于避免重复注册。`blog` 表是系统中最核心的内容表，保存标题、正文、分类、标签、作者、原文链接、摘要、浏览量、点赞数、收藏数和平均分等信息，能够同时支撑内容展示、搜索和推荐。`comment` 表用于保存评论文本、评分、评论人信息、文章关联关系和评论时间，是评论区与评分功能的共同数据基础。

`notification` 表用于支撑消息通知功能，保存接收用户、触发用户、通知类型、通知摘要、关联文章和已读状态等信息。虽然表结构中为收藏类通知预留了类型值，但当前项目中已确认的主链通知仍以点赞和评论为主，因此论文写作时应以已证实行为为主，不将预留值直接写成系统现状。

### 3.4.3 行为记录表设计

为了支持推荐和个人中心回显，系统将行为数据拆分为三类表。`user_like` 表专门保存点赞关系，并通过唯一约束避免同一用户对同一文章重复点赞。`user_action` 表统一保存收藏、待读和拉黑三类动作，通过动作类型字段区分具体业务语义，兼顾了结构统一和功能扩展。`visit_log` 表记录谁在何时阅读了哪篇文章以及停留多久，是阅读历史、累计阅读时长和画像构建的直接数据来源。

这种拆分方式的好处在于，点赞、动作和阅读日志虽然都属于行为数据，但更新频率、业务意义和后续使用方式并不完全相同。将其分开存储后，推荐模块可以按需要分别读取不同类型信号，而个人中心页面也能更直接地进行分类展示。

### 3.4.4 数据库边界说明

在数据库脚本中，还存在 `recommendation` 和 `user_behavior` 两张表。前者用于保存推荐结果和推荐理由，后者用于统一记录浏览、收藏和拉黑等行为。然而从当前实际业务链看，推荐主链主要依赖 `visit_log`、`user_like` 和 `user_action` 获取用户行为，并未确认形成基于 `recommendation` 表的完整落库与回读闭环；`user_behavior` 表也未表现为当前主业务链的核心使用表。

因此，在论文总体设计章节中，这两张表只能作为“数据库中存在但未作为当前主链核心展开的结构”进行边界说明，不能写成系统当前已经正式投入使用的核心表。

## 3.5 个性化推荐模块总体设计

### 3.5.1 推荐目标与输入依据

推荐模块的总体目标不是追求复杂模型实验，而是在真实博客场景中基于现有行为信号生成可运行、可解释的推荐结果。当前系统将浏览记录、阅读时长、点赞、收藏、评分和拉黑等行为纳入推荐输入，并通过标签信息将内容特征与行为特征连接起来。与只根据文章热度排序相比，这种设计更适合技术博客场景中长期阅读兴趣逐步积累的特点。

同时，系统在配置中为阅读时长权重、结果数量、同作者上限、同标签上限和推荐理由开关等提供了参数入口，使推荐模块具备一定可调性。这样的设计更符合工程项目的可维护性要求。

### 3.5.2 用户兴趣画像设计

用户画像服务以文章标签为主要建模对象。系统会根据用户的浏览、阅读时长、点赞、收藏和评分行为，累计不同标签在用户行为中的权重，并据此得到兴趣标签摘要和雷达图展示结果。其中，阅读时长并不是简单地作为一次访问记录，而是被用来区分浅层浏览和深度阅读；评分则作为显式反馈信号，用于增强画像对用户认可程度的区分能力。

在冷启动判断方面，当用户行为积累不足时，系统不会强行生成复杂画像，而是允许推荐链路直接回退到热门内容兜底。这种处理方式虽然保守，但能够保证推荐模块在行为数据不足时仍保持稳定可用。

### 3.5.3 推荐召回策略设计

当前系统采用多路召回组合的方式生成推荐候选。第一路是协同过滤召回，用于从其他用户行为关系中寻找可能感兴趣的内容；第二路是标签召回，用于根据当前用户较高权重标签匹配内容；第三路是热门兜底召回，用于在行为不足或候选不够时补足推荐结果。三路结果合并后进入后续重排阶段。

这种召回设计与项目现实条件相匹配。一方面，系统已有的行为数据规模和实现复杂度更适合采用可解释的规则与轻量推荐方式；另一方面，多路召回也使系统在不同用户状态下都能获得可展示的结果，不会因为单一路径失效导致推荐页不可用。

### 3.5.4 推荐重排与解释设计

在候选结果形成后，系统通过多样性重排控制推荐列表中同作者和同主标签的过度集中，避免推荐结果出现明显重复。与此同时，系统在 `Blog` 实体中设置了 `recommendSource`、`recommendReason` 和 `hotScore` 等仅在运行链路中使用的字段，用于向前端说明推荐来源、展示推荐理由，并在热门计算过程中传递中间分值。这些字段不直接落库，而是作为推荐过程中的动态信息存在。

系统还在配置中保留了 AI 精排相关参数，但当前 `recommend.ai-rerank-enabled=false`，说明 AI 精排策略仍处于扩展保留状态，而不是当前默认主链。对于论文写作而言，可以将其写为“系统预留后续增强接口”，不能写成“系统已启用大模型重排序”。

### 3.5.5 推荐模块边界说明

综合当前实现情况，系统的推荐模块可以被准确概括为“基于用户行为和标签画像的混合推荐设计，并结合热门兜底和推荐理由展示增强可用性与可解释性”。这一表述能够覆盖当前真实主链，同时避免夸大系统能力。

需要明确的是，当前项目未开展推荐效果的量化评估，也未形成推荐结果持久化落库与回读的完整闭环，因此本章只从总体设计和工程实现角度说明推荐模块的结构与数据流，不对算法效果做超出证据范围的结论性判断。

# 第4章 系统详细设计与实现

## 4.1 用户账户与基础访问控制功能实现

系统的账户功能由注册、登录、资料修改、头像上传和个人数据展示几个部分组成。前端入口主要分布在登录页、注册页和个人中心页。用户首次进入系统时，需在注册页填写用户名、密码和昵称等基础信息，前端将数据提交至注册接口后，由后端完成用户名重复校验和密码加密存储。登录成功后，用户信息会保存在前端本地存储中，用于后续页面访问和状态回显。

在访问控制方面，系统没有引入完整的安全框架，而是采用更符合当前项目复杂度的基础访问控制方式。前端通过路由守卫限制未登录状态访问首页、详情页和个人中心等核心页面，后端则在部分修改与删除操作中结合当前用户身份进行校验。这样的实现方式虽然不属于完整权限体系，但已经能够满足当前博客系统对账户入口和受限页面访问的基本要求。

个人中心页还承担资料维护和个人数据展示功能。用户可修改昵称、头像和密码，并在修改成功后同步更新前端缓存中的用户信息。除资料维护外，个人中心还会调用用户统计、兴趣摘要和雷达图接口，将累计阅读时长、兴趣标签摘要和可视化结果展示给用户。由此，用户模块不仅承担身份入口功能，也构成后续行为数据展示的统一入口。

## 4.2 博客信息录入、查询与管理实现

博客录入功能主要由首页中的文章发布入口完成。用户在首页打开发布表单后，可填写标题、正文、标签、分类、原文链接和摘要等内容，并可选择上传封面图片。提交后，前端将文章数据发送至博客新增接口，由后端博客服务统一完成保存。若用户未手动填写摘要，系统会根据正文自动截取生成基础摘要；若用户主动使用 AI 分析，则先调用 AI 接口生成摘要和标签建议，再由用户确认后提交正式文章。

文章查询与管理主要体现在列表展示、关键词搜索、详情浏览和编辑删除几个环节。首页通过文章列表接口获取全部文章，并结合关键词搜索接口实现快速检索；用户点击文章后进入详情页查看正文、标签、评分和相关推荐。若当前用户是文章作者，详情页还会显示编辑入口；若当前用户是作者或超级管理员，则允许执行删除操作。删除和修改最终均通过博客服务完成，并将结果反馈到前端列表或详情页。

从实现效果看，博客模块不仅完成了传统的增删改查，还与推荐、阅读记录和 AI 辅助形成联动。文章标签、摘要和正文既服务于页面展示，也构成后续画像与相关推荐的内容侧基础数据，因此该模块是全文实现部分的重要基础。

## 4.3 阅读记录与个人中心实现

阅读记录功能围绕文章详情页展开。用户进入详情页时，系统会先读取文章详情，并同步增加阅读量和写入浏览日志。当前实现中，阅读行为不是简单的一次点击记录，而是进一步跟踪页面停留时长。当前端检测到用户离开页面且停留超过阈值后，会将本次阅读秒数回传给后端，由后端更新阅读日志中的时长信息。这样，浏览历史和阅读时长便同时沉淀到行为数据中。

个人中心中的浏览历史页负责回显用户已经阅读过的文章记录，累计阅读时长则以统计数值方式展示在概览区域。由于阅读记录与文章内容之间建立了关联，系统不仅能展示“看过什么”，还能为后续兴趣标签归纳提供基础。与只记录点击次数的做法相比，当前实现更强调阅读行为的持续性信息，这也是推荐模块能够使用阅读深度信号的重要前提。

个人中心本身承担的是多项个人功能的统一承载。除浏览历史外，它还集中提供我的文章、我的点赞、收藏列表、待读列表、黑名单列表、消息中心和资料设置等入口。这样的实现使用户不需要在多个页面之间来回切换，就能回看自己的内容产出和互动结果，也使第4章后续多个模块能够在同一页面体系中展开说明。

## 4.4 点赞、收藏、待读、拉黑、评论评分与通知实现

互动功能主要集中在文章详情页。对于点赞操作，系统设置了专门的点赞接口和点赞关系表，用于记录当前用户是否对指定文章点过赞，并在详情页或个人中心中回显状态。收藏、待读和拉黑三类动作则通过统一动作接口处理，并在数据库中以动作类型区分具体语义。这种设计在保证页面交互统一的同时，也降低了后端接口数量和数据结构复杂度。

其中，拉黑功能除了记录用户不感兴趣的文章外，还会参与后续文章列表、搜索结果、热门内容和推荐结果的过滤，因此它不只是一个个人书架状态，更是推荐链路中的负向信号。收藏和待读则主要用于书架管理和兴趣表达，前端在个人中心中通过不同列表进行分类展示。

评论与评分共用同一套提交流程。用户在详情页既可以输入评论文本，也可以只提交星级评分。评论记录写入评论表后，系统会同步计算或更新文章平均分；当评论被删除时，文章均分也会被重新计算。当前实现中，评论回复主要通过前端预填内容方式体现，并未形成独立的楼中楼结构，因此论文写作时应将其表述为“评论交互”而非“多层评论系统”。

在当前本地工作副本中，评分不仅用于文章均分展示，也会进一步作为显式反馈信号参与兴趣画像计算。这意味着评分功能在实现层面同时承担了“内容评价”和“偏好表达”两类作用，但仍不应被拔高为完整评价模型或推荐效果评估体系。

通知功能与互动行为形成联动。当前已确认的通知触发场景主要包括点赞和评论两类：当用户对他人文章点赞或发表评论时，系统会向文章作者写入通知记录。消息中心支持获取未读数、拉取消息列表、单条已读和全部已读。需要特别收紧的边界是，虽然通知表结构中预留了收藏类型字段，但当前代码未发现收藏触发通知的主链实现，因此不能在正文中写成系统现状。

## 4.5 数据概览与画像展示实现

当前系统的数据展示并不面向后台运营分析，而是面向普通用户的个人概览与兴趣回看。个人中心会展示累计阅读时长、文章数量、互动结果等基础统计信息，使用户能够快速了解自己在系统中的使用情况。与传统博客系统相比，这部分实现增强了用户对自身行为结果的可见性。

画像展示则建立在阅读、点赞、收藏、评分和标签信息的整合基础上。后端根据用户的行为记录汇总兴趣标签权重，并将其整理为文字摘要和雷达图数据返回给前端。前端再利用图表组件将标签分布可视化展示出来，使用户能够直观看到自己的兴趣方向。这里的画像结果主要服务于“行为总结”和“推荐解释”，不应被拔高为独立画像平台。

从实现角度看，这部分功能的意义在于把推荐模块使用的内部行为信号转化为用户可理解的展示结果，从而增强系统的透明度和可解释性。也正因为如此，第4章可以将其作为推荐模块与个人中心之间的重要连接点展开说明。

## 4.6 个性化推荐与 AI 辅助功能实现

个性化推荐功能主要体现在首页“猜你喜欢”和详情页“相关推荐”两个位置。首页推荐接口会根据当前用户的阅读、点赞、收藏、评分和拉黑等行为信号构建兴趣画像，再交由推荐引擎生成推荐结果；详情页相关推荐则更多依据当前文章标签，补充内容相近的候选文章。对于行为信号不足的用户，系统通过热门内容进行兜底，避免推荐区为空。

从推荐链路看，当前实现并不是单一路径推荐，而是采用协同召回、标签召回和热门兜底组合方式形成候选，再通过多样性重排控制同作者和同主标签内容的过度集中。推荐结果在返回前还会附带推荐来源和推荐理由，前端可据此展示“为什么推荐这篇文章”。这种实现方式更加贴近博客场景中的可解释需求，也便于用户理解推荐结果与自身兴趣之间的关系。

AI 辅助功能目前主要服务于文章录入环节。用户在发布文章前可主动调用 AI 分析接口，请求生成摘要和标签建议；若外部 AI 调用失败，后端会返回降级结果，以本地截取摘要和占位标签保证功能可用。需要注意的是，AI 精排虽然在配置中保留了开关和数量限制，但当前默认关闭，因此正文只能写成“系统预留 AI 增强能力”，不能写成“推荐结果已经由大模型主导重排”。

此外，推荐相关的 `recommendSource`、`recommendReason` 和 `hotScore` 等字段在实体中只作为运行期动态信息使用，并未形成推荐结果落库回读的完整闭环。因此，本系统当前的推荐实现应准确表述为“运行期生成并展示推荐结果”，而不是“建立推荐结果持久化平台”。

## 4.7 关键页面与前后端交互实现

从页面交互角度看，系统最关键的五个页面分别是登录页、注册页、首页、文章详情页和个人中心页。登录页和注册页负责用户进入系统的起点；首页承载文章列表、搜索、热门内容、推荐区域和发布入口；详情页承载阅读、互动、评论评分和相关推荐；个人中心则集中承载消息、书架、历史、资料和画像等个人功能。这样的页面分布使系统在较少页面数量下完成了较完整的业务覆盖。

前后端交互方面，系统以前端 Axios 请求和后端 REST 风格接口为主。页面的主要状态变化都能够对应到明确的后端接口，例如注册登录对应用户接口，文章录入和搜索对应博客接口，点赞和收藏对应互动接口，消息页对应通知接口，推荐区和画像页对应推荐与用户统计接口。这样的接口组织使前端状态回显逻辑相对直接，也便于在论文中以“页面—接口—服务—数据表”的链路方式说明实现过程。

需要强调的是，当前章节对“实现”的说明基于真实静态代码链路和数据库结构，而不是基于线上环境性能验证或自动化测试结果。因此，本章能够确认的是“功能链路存在并具有明确实现路径”，而不能超出证据范围宣称系统已经完成完整运行稳定性验收。这样的收束更符合本科软件工程项目论文对真实性的要求。

# 第5章 系统测试与结果分析

## 5.1 测试环境

本章测试以本地运行环境为基础开展，目标是验证系统核心业务链条是否能够按设计完成基本闭环，而不是进行性能评估或推荐效果量化实验。测试所使用的代码副本为 `D:\Project\blog\blog-new`，前端采用 Vue 3、Vite、Element Plus 和 Axios，后端采用 Java 21、Spring Boot 4.0.3、MyBatis 和 MySQL。数据库环境中已存在基础业务数据，可支撑注册、发文、互动、通知与推荐等接口联调。

在本轮验证中，前端已完成生产构建，后端已完成 `contextLoads()` 基础测试，并实际对注册、登录、发文、搜索、详情、阅读时长、点赞、收藏、待读、拉黑、评论评分、通知和推荐接口进行了联调。由于本课题并未进行并发测试和算法量化实验，因此本章只讨论功能正确性、流程可用性、异常输入处理和推荐逻辑的定性表现。

## 5.2 功能测试方案

结合当前项目材料和本科论文的真实性要求，本章采用“构建验证 + 接口联调 + 数据库回查 + 定性分析”的测试方案。相关 Web 测试研究表明，功能测试、测试用例组织、测试充分性判断和执行结果回查，是 Web 系统测试中较为常见且具有工程价值的切入点[18][19][20][21][22]。具体而言，本文首先通过前端生产构建和后端基础测试确认项目在当前环境下具备基本运行条件；其次，围绕系统主业务流程设计接口级测试用例，验证账户、内容、互动、通知和推荐等功能能否正常返回；最后，通过数据库回查方式验证部分关键业务结果是否真正落库，从而提高测试结论的可信度。

测试对象主要覆盖四类内容。第一类是账户与内容功能，包括注册登录、文章发布、搜索和详情查看；第二类是行为与互动功能，包括阅读时长、点赞、收藏、待读、拉黑、评论和评分；第三类是通知与权限边界，包括消息已读和无权限删除拦截；第四类是推荐相关功能，包括推荐结果返回、推荐理由展示、画像摘要回显和黑名单过滤。该测试方案能够较好对应本系统的真实功能结构，同时避免写入无依据的性能类结论。

## 5.3 关键业务流程测试结果

从账户与内容流程看，本轮测试已成功使用新建测试用户完成注册和登录，并得到有效用户标识，说明用户入口链路能够正常闭合。随后，测试使用 ASCII 标识标题发布了一篇新文章，并通过搜索接口命中该文章，说明发布与检索链路在当前环境下能够正常工作。进一步调用详情接口后，能够读取到文章标题、正文和阅读量信息，说明文章详情查询链路是可用的。

从阅读记录与互动流程看，测试用户访问文章详情后，再提交阅读时长，数据库 `visit_log` 中可查询到对应记录，用户统计接口返回的总时长也发生了变化，说明阅读记录与时长累计链路能够成立。与此同时，点赞、收藏、待读和拉黑接口均返回成功，且回查结果能够确认点赞状态、收藏列表和待读列表均已更新，文章维度的拉黑过滤在列表接口中也得到了体现。

从评论、评分、通知与权限边界看，测试用户对既有文章提交评论和评分后，评分接口能够返回 `4.0`，数据库评论表也能查到对应测试记录，说明评论评分链路是可用的。对同一文章执行点赞和评论后，目标作者的未读通知数由 `0` 增加到 `2`，随后执行单条已读和全部已读操作后，未读数归零，说明通知联动和消息状态处理符合预期。此外，测试用户尝试删除他人文章时，接口明确返回“权限不足”，说明基础访问控制和局部权限校验能够发挥作用。

综合本节结果可以看出，系统在账户、内容、互动和通知等核心流程上已经具备较稳定的功能正确性。需要指出的是，本轮测试主要通过接口联调和数据库回查完成，尚未补齐前端页面截图与完整端到端点击记录，因此当前结论应理解为“系统核心业务链具备可验证可用性”，而不是“所有页面场景均已全面验收”。

## 5.4 推荐逻辑验证与结果分析

推荐模块的验证重点不是准确率或召回率，而是接口是否返回有效结果、结果是否带有可解释信息，以及用户行为是否会影响推荐上下文。测试用户在完成少量阅读、点赞、收藏和评分操作后，请求个性化推荐接口，系统返回了 30 条推荐结果，且首条结果带有 `recommendSource` 和 `recommendReason` 字段。这说明推荐链路在当前环境下能够稳定返回可展示的结果，而不是空列表或不可解释的裸数据。

进一步观察推荐来源可以发现，当前测试用户的首条推荐来源为 `hot`，推荐理由也体现为热门内容说明。这一结果说明在轻量行为用户场景下，系统仍然会明显依赖热门兜底策略，以保证推荐页可用性。该现象与系统的总体设计是一致的，即在行为数据不足或画像信号不够强时，优先采用热门内容稳定输出，而不是强行给出不可靠的个性化结果。

在用户画像侧，测试用户的画像摘要中已经能够回显“前后端分离、博客系统、内容推荐”等标签，这表明用户近期交互过的文章标签已经进入画像汇总链路。同时，黑名单过滤验证显示，被拉黑文章不会继续出现在用户视角的文章列表中，说明负向反馈已经能够影响内容展示结果。综合来看，本系统的推荐逻辑在当前阶段已经具备“可返回、可解释、可受行为影响”的基本特征，但本章只对其进行定性分析，不对推荐优劣作任何量化判断。

## 5.5 存在的问题与改进方向

虽然本轮测试证明了系统核心功能链条基本可用，但仍存在几个需要在后续完善的方面。第一，当前自动化测试覆盖范围较小，后端仅确认了基础上下文加载测试通过，尚未形成覆盖注册、发文、互动和推荐接口的自动化测试集。第二，前端虽然能够成功完成生产构建，但构建日志中已出现大体积 chunk 提示，说明后续仍可从资源拆分和按需加载角度继续优化。

第三，本轮测试主要依赖接口联调和数据库回查，前端页面截图和端到端交互记录仍需进一步补齐，这意味着第5章目前更偏“工程验证”而非“界面验收”。第四，本次运行与联调基于用户指定的本地工作副本完成，而该副本相对远程仓库仍存在本地改动，因此后续若要形成最终答辩稿，还需要在正文或附加说明中明确测试所依据的是哪一份代码快照，避免不同代码快照之间的事实边界混淆。

# 第6章 结论与展望

## 6.1 工作总结

本文围绕一个真实技术博客项目，完成了“个性化技术博客推荐系统”的设计与实现整理工作。与只关注单一展示页面的博客项目不同，本系统围绕账户管理、博客录入与查询、阅读记录、互动反馈、消息通知、个人中心、画像展示和个性化推荐构建了较完整的业务闭环。结合前文内容，可以将本文已完成的主要工作概括为以下几个方面。

第一，完成了系统需求分析与总体设计。本文在明确项目真实边界的前提下，对系统建设目标、功能需求、非功能需求和业务流程进行了梳理，并结合前后端分离结构、业务模块划分、数据流组织和数据库设计，对系统总体架构进行了说明。第二，完成了系统关键功能的实现分析。围绕账户与资料管理、博客录入与管理、阅读记录与阅读时长、点赞收藏待读拉黑、评论评分、通知联动、画像展示和推荐链路，本文给出了从页面到接口、从服务到数据表的实现说明。第三，完成了系统测试与结果分析。通过构建验证、接口联调、数据库回查和推荐逻辑定性观察，验证了系统核心业务链的可用性，并对当前结果进行了克制的工程分析。

就项目本身而言，系统已经具备较完整的博客平台基础能力，并在此基础上加入了行为采集、画像展示和混合推荐能力。推荐部分没有走复杂模型训练路线，而是采用更符合当前工程条件的混合推荐思路，将浏览、阅读时长、点赞、收藏、评分和拉黑等信号纳入兴趣画像，并结合热门兜底和推荐理由展示，提高推荐结果的可用性与可解释性。AI 能力则主要体现在文章录入阶段的摘要与标签生成，体现为对已有业务流程的辅助增强，而不是替代核心业务逻辑。

## 6.2 不足分析

尽管本文完成了系统的主要功能整理和工程验证，但从真实项目边界出发，当前系统仍存在较为明确的不足。

第一，性能测试与推荐量化评估尚未开展。本文没有进行并发测试、吞吐量测试和响应时间实验，也没有计算推荐准确率、召回率、NDCG、MAP、AUC 和 F1 等算法指标。因此，本文对系统的评价只能停留在功能正确性、流程可用性和推荐逻辑定性表现层面，而不能扩展为性能优化或算法效果结论。第二，自动化测试覆盖范围仍然有限。当前后端仅确认了基础 `contextLoads()` 测试通过，虽然接口联调和数据库回查已经补充了较多真实证据，但距离完整的自动化测试体系仍有明显差距。

第三，页面级证据仍有补强空间。当前第5章已经具备构建验证、接口联调和数据库回查材料，但前端页面截图和完整端到端交互记录仍不充分，因此全文在“实现可视化证据”层面还有进一步加强的空间。第四，推荐模块虽已形成可运行主链，但 AI 精排并未作为默认主链启用，`recommendation` 表和 `user_behavior` 表也未形成稳定的主业务闭环，因此系统当前仍应被理解为“以混合推荐为主、保留 AI 增强接口”的实现状态，而非更大规模的智能推荐平台。

此外，当前运行验证依赖本地工作副本完成，而该副本相对远程仓库仍存在本地改动。这意味着后续若要形成答辩终稿或对外展示版本，还需要继续做好代码快照、测试记录和论文表述之间的一致性校验，避免不同快照之间的事实边界混淆。

## 6.3 后续展望

在后续工作中，系统仍有几个较为明确且可落地的改进方向。

第一，可以继续补强测试体系。包括增加接口级自动化测试、补充页面级截图和端到端交互验证，并对当前已发现的历史展示、行为回显和前端交互一致性问题进行进一步确认。第二，可以在真实数据积累更充分的前提下，补做推荐效果的定性对比和更规范的人工评估，但这类工作应建立在可追溯测试方案和真实数据样本之上，而不是简单追加实验指标。第三，可以在保持主链稳定的基础上，进一步探索 AI 精排、推荐理由优化和标签语义增强等能力，但这些增强应继续作为扩展方向推进，而不是在证据不足时提前写成既有成果。

第四，可以继续优化系统的工程化程度。例如，对前端较大的构建包进行拆分，对页面返回链路和个人中心展示逻辑做进一步整理，并逐步形成更完善的运行与部署说明。总体来看，本文所完成的工作已经为后续增强推荐能力、补强测试体系和完善工程结构打下了基础，但这些改进都应在保持真实性和工程边界清晰的前提下推进。
