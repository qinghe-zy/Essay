# 第2-5章写作边界说明

## 第2章 需求分析
- 可以写：建设目标、功能需求、非功能需求、业务流程、可行性分析。
- 必须基于：项目事实卡片、页面路由、接口链路、SQL、开题报告中的题意边界。
- 不能写：计划中的算法实验、完整权限体系、独立后台平台。
- 易失真点：把计划需求写成系统现状。
- 降 AI 味：多写真实操作链路和工程约束，少写空泛意义。

## 第3章 系统总体设计
- 可以写：技术栈、架构设计、模块设计、业务流程、数据库设计、推荐模块总体设计。
- 必须基于：源码结构、配置、控制器/服务/Mapper、SQL、推荐主链代码。
- 不能写：`recommendation`、`user_behavior` 是当前核心生产链；AI 精排已启用。
- 易失真点：把预留表、预留代码、默认关闭能力写成当前核心设计。
- 降 AI 味：按真实调用链和字段职责写，不堆概念。

## 第4章 系统详细设计与实现
- 可以写：页面、接口、服务逻辑、状态变化、前后端交互、关键模块实现。
- 必须基于：前端页面源码、控制器、服务层、Mapper、表结构、接口调用链。
- 不能写：README 扩写、无证据页面、无证据接口、无证据功能。
- 易失真点：把 README 描述当实现，把局部管理员豁免写成完整权限模块。
- 降 AI 味：优先写“输入 -> 调用 -> 数据变化 -> 页面反馈”。

## 第5章 系统测试与结果分析
- 可以写：测试环境、功能测试、流程测试、异常测试、推荐逻辑定性验证、存在问题。
- 必须基于：测试材料包、源码逻辑、页面行为、接口联调流程。
- 不能写：性能测试、推荐量化评估、实验指标。
- 易失真点：为了看起来完整而硬写并发与算法数据。
- 降 AI 味：按“测试目的 -> 测试步骤 -> 预期结果 -> 实际表现”写。
